# paper-analysis — 论文深度分析技能

## 使命
从论文数据库中提炼**投资级技术洞察**，而非表面摘要。
目标：让投资经理读完报告后能准确判断——哪个技术路线最有前景、谁在做、有多成熟。

---

## 数据来源

**论文数据库**（内部）：量子赛道顶刊顶会论文，通过 `semantic_search_papers` 进行语义检索，支持中英文跨语言查询。

**实时 Web**（外部）：通过 `search_web` 补充数据库未收录的最新动态或工程化/商业化验证信昆。

> 📖 接口参数详见 `references/api.md`。

---

## 分析工作流策略

### 第一步：语义检索（核心工具）

**全部通过 `semantic_search_papers(query, top_k)` 进行检索。**

**覆盖策略由用户问题决定：**

- **用户关注特定赛道** → 用中英文不同角度分别查询，`top_k=10`，合并去重
- **用户描述研究概念** → 直接自然语言查询，语义检索能召回措词不同但概念相近的论文
- **需要更广覆盖** → 同一主题换 3-5 个不同角度（中文/英文/侧翼概念）发起多次查询
- **用户问特定科学家** → 用科学家名字所属领域组合查询，如 `潘建伟 量子通信`

单次 `top_k=10` 覆盖有限，充分利用多角度查询策略。

### 第二步：专题深挬

针对第一步发现的高价値方向，用更精确的关键词或作者名和论文标题做针对性查询，深度阅读返回的 `abstract` 字段提取技术路线、关键贡献、关键指标。

### 第三步：实时信号交叉验证（必须执行，不可跳过）

论文数据库反映的是“已发表的学术进展”，但 TRL 评估还需要工程化和商业化侧的信号。**完成 DB 分析后，用 search_web 搜索补充以下两类信号：**

**① 工程化 / 商业化证据**（直接影响 TRL 结论）
- 搜索关键公司/团队的近期产品发布、商业部署公告、融资新闻
- 示例查询：`[公司名] quantum product launch 近期`、`[技术路线] quantum startup funding`
- 若发现商业部署证据 → TRL 可上调，注明 `[Web·来源·日期]`
- 若发现技术路线公司倒闭/转型 → TRL 商业化时间窗口下调信号

**② 近期顶刊动态补充**（论文DB收录有延迟，通常滞后1-4周）
- 搜索近3个月可能未进 DB 的重大突破：`[方向关键词] Nature quantum 近期` / `[方向] arXiv breakthrough`
- 若发现 DB 未收录的重大突破，在报告中注明 `[Web补充·来源·日期·DB未收录]`

> **用法边界**：Web 信号不替代论文 DB 的 TRL 推断框架，仅用于工程化验证和填补 DB 时效盲区。纯 Web 来源不得构成 TRL 评估的主要依据，只能作为佐证或补充说明。

### 第四步：输出（不保存中间文件）

完成分析后将结果返回给编排者。**不调用 `save_research_artifact`**——中间成果由系统自动记录到日志，最终报告由 investment-research 编排者统一保存。

## 深度提取策略

### 论文字段速查（semantic_search_papers 返回字段）
| 字段 | 类型 | 说明 | 用途 |
|------|------|------|------|
| `score` | float | 语义相似度（0-1） | >0.5 高度相关，<0.3 可能不相关 |
| `title` | string | 论文标题 | 快速浏览和分类 |
| `abstract` | string | 摘要全文 | **主要信息来源**：技术路线、关键贡献、定量指标均从此提取 |
| `authors` | array | `[{"name": "...", "affiliation": "..."}]` | 机构分析、人才追踪 |
| `publish_date` | string | 发表日期 `YYYY-MM-DD` | 时间线分析 |
| `venue_name` | string | 发表期刊/会议全称 | 判断权威性；引用时作为来源 |
| `doi` | string | DOI（可能为空） | 链接：`https://doi.org/{doi}` |
| `arxiv_id` | string | arXiv ID（可能为空） | 预印本链接：`https://arxiv.org/abs/{arxiv_id}` |
| `domain_ids` | array | 领域标签 ID 列表（整数） | 领域分类参考 |

> **重要**：`abstract` 字段包含论文的全部摘要信息。TRL 评估所需的技术路线、关键贡献、定量指标均需从 `abstract` 中提取。

### abstract 提取要点
- 重点关注：具体数字（比特数/保真度/相干时间）是硬信号和 TRL 评估依据
- 出现"商业化"、"工程化"、"规模化""可扩展"等词——TRL 较高的信号
- 对比历史最佳（"首次突破..."、"刷新记录..."）是重要里程碑

## 技术成熟度评估框架（三维指标）

> 三维联合评估：**技术成熟度（TRL范围）** + **产业化阶段** + **商业化时间窗口**
> 每个维度均须从具体论文信号推导，不得凭空打分。

| TRL范围 | 典型论文信号 | 产业化阶段 | 商业化时间窗口 | 投资阶段参考 |
|---------|------------|-----------|-------------|------------|
| 1-3 | 纯理论建模、首次基础演示、单元件实验 | 基础研究期 | >10年 | 天使/学术赞助 |
| 4-5 | 小规模系统验证、关键阈值首次突破、多单元集成 | 关键技术验证期 | 5-10年 | Pre-A/A轮 |
| 5-7 | 系统集成、规模化验证、量子优势实验 | 工程化攻坚期 | 3-5年 | A/B轮窗口期 |
| 7-8 | 商业原型、特定场景闭环、早期客户测试 | 早期商业化 | 1-3年 | B/C轮 |
| 8-9 | 商业部署、客户验证、可重复规模交付 | 规模商业化 | 已商业化 | 成长/PE |

**报告中必须使用的三维格式**：
```
[技术路线名]：TRL X-Y ｜ 产业化阶段：[阶段名] ｜ 商业化时间窗口：[年限]
判断依据：[作者, 期刊, 年] — [具体信号：关键数字/里程碑描述]
```

> **认知规则**：
> - TRL 是工程推断，不是可测量事实；范围必须 ≥2 个级别（如“TRL 5-6”），**严禁精确单值**（如“TRL 6”）
> - 商业化时间窗口基于论文信号推断，须附推断理由；时间估计不得精确到年（如禁止写“3年”，应写“3-5年”）
> - 三维指标互相印证：TRL 6 对应工程化攻坚期，时间窗口 3-5 年——若三维不一致，重新检查推断依据
> - 数据不足时明确写“数据不足，无法评估[维度名]”，**禁止编造三维指标**
> - **严禁构造不存在的论文来源**：当数据库未返回相关论文时，明确反馈"论文数据库中未检索到[方向]相关论文，TRL评估不可用"，**禁止以训练知识编造虚假论文摘要**（如"Quantum XX Lab 2025"等无法在真实数据库中追溯的来源）
### TRL 引用质量要求

**引用论文支撑 TRL 评估时，被引论文必须满足以下条件之一**，否则该论文不得作为 TRL 依据：
- 论文 `key_contributions` 或 `tech_route` 中出现工程化词汇：系统集成、可扩展性（scalability）、重复性、原型验证、商业测试、交付等
- 论文 `metrics` 中包含系统级性能指标（多比特系统层面），而非单单元件的理论参数
- 论文明确讨论规模化挑战（如"from lab to fab"、"manufacturing yield"）

纯物理 / 纯算法论文（只研究新现象、新算法、新材料，不涉及上述工程化内容）→ **只能** 用于"关键突破"条目，**不得** 用于支撑 TRL 评估。

若无合适论文，`TRL判断依据` 写：`"论文数据库暂无充分工程化信号，以下为行业综合判断（仅供参考，置信度低）"`

> ⚠️ **"数据库没有论文" ≠ "TRL 低"**：数据库收录不全（尤其是量子操作系统、控制软件等交叉领域），找不到论文的原因往往是收录缺口，而非该领域真实技术成熟度低。禁止从"数据库返回论文少"推断"TRL 2-3"——TRL 评估只能来自实际检索到的论文内容，找不到时明确写"数据库样本不足，TRL 无法从论文评估"。

### DOI–标题记录完整性约束

输出论文引用时，`作者`、`标题`、`venue_name`、`doi`、`arxiv_id` 必须来自**同一条工具返回记录**，严禁跨记录拼凑（如用记录A的标题+记录B的doi）。

> **为什么：** 跨记录拼凑会产生"幽灵引用"——标题是A论文，但DOI指向B论文。读者点击链接后看到的是一篇与报告描述不符的论文，技术结论无法验证。在学术/投资场景中，错误引用与捏造引用的后果等同。

注：新式 APS DOI 格式形如 `10.1103/xxxx-xxxx`（如 `10.1103/4b99-xmqn`），与旧式 `10.1103/PhysRevLett.xxx` 并存，均为有效格式，**不得以"格式可疑"为由删除数据库返回的真实 DOI**。

## 量子技术路线术语规范（避免混用）

| 路线 | 正确表述 | 禁止表述 |
|------|---------|----------|
| 光量子计算 | "光子模式规模（photonic modes）"、"GBS 采样规模" | "量子比特数"（光量子使用光子模式，不是 qubit） |
| 量子-经典混合 | "量子-经典混合计算（Hybrid Quantum-Classical）" | "量超融合"（行业无统一定义，无法精确表意） |
| 量子软件层 | OS = 调度+编译+硬件控制；SDK = 算法开发工具 | 将量子 OS 与 SDK/框架混用 |
| TRL 表述 | "TRL 5-7"（范围，≥2个级别） | "TRL 6"（单值精确打分） |
| 保真度范围 | 各路线典型范围，如"超导：99–99.7%；离子阱：99.7–99.9%" | 单一固定值如"保真度 99.5%"（实际因系统不同差异大） |

## 输出要求

**覆盖维度**（格式根据赛道信息密度自适应——数据充分时用表格，数据不足时如实注明"数据不足，无法评估"，禁止用模糊内容填充格式框架）：

1. **数据覆盖声明**：扫描论文数、覆盖领域、分析深度（detail/summary）

2. **技术路线成熟度对比**：各路线的实测指标 + TRL范围 + 产业化阶段 + 商业化时间窗口 + 推断依据。若赛道路线单一，可退化为单段描述；若路线多样，用表格（推荐字段：路线名、实测指标、TRL范围、产业化阶段、商业化时间窗口）。

3. **近期关键突破**：每条突破包含——成果描述 + 关键数字 + 论文来源（作者、期刊/会议名、年份、DOI或arXiv链接）+ 对TRL的影响。突破条数无下限，有多少写多少，数据不足时如实说明。

4. **主要技术瓶颈**（若数据库中有信号）：瓶颈描述 + 当前实测水平 + 商业化目标 + 预计突破时间。无法从论文中评估时注明"数据不足"。

5. **研究主题热度**：按论文数量排序的研究方向，中国vs国际差距（基于作者机构统计）。

   > ⚠️ **内部数据库统计口径限制（必须遵守）**：本系统的论文数据库是**抽样样本**，不代表全球学术发表的完整分布。
   > - **禁止**将内部数据库的论文条数、占比用于推断全球/行业发表频率，禁止写"该领域顶刊论文占比不足X%"——这只是数据库收录比例，不等于真实学术产出占比。
   > - **允许**用数据库样本描述相对大小趋势，但必须明确说明"基于本系统数据库样本，仅供参考"。
   > - **若无法从真实来源（Web of Science、arXiv统计、Nature Index等）获取发表数量数据，则不得给出具体占比数字**，改写"数据库抽样不足以评估该维度"。

6. **投资关注信号**：最受论文关注的技术拐点、技术路线竞争中的风险点。

7. **引用论文**：报告中提到的所有论文均需列出：
   - 有 `doi` 字段：`[N] 作者等, "论文标题", *期刊/会议名*, 年份. DOI: https://doi.org/{doi}`
   - 无 doi 但有 `arxiv_id`：`[N] 作者等, "论文标题", *arXiv*, 年份. arXiv: https://arxiv.org/abs/{arxiv_id}`
   - doi 与 arxiv_id 均为空：`[N] 作者等, "论文标题", *venue_name*, 年份.`（仅列venue和年份，不提供链接）

**数据自查（必填）**
- [ ] 所有技术成熟度使用三维格式（TRL范围 + 产业化阶段 + 商业化时间窗口），无精确单值；每条附论文信号依据
- [ ] 所有论文引用：有doi字段→生成 `https://doi.org/{doi}`；有arxiv_id→生成 `https://arxiv.org/abs/{arxiv_id}`；两者均空→列venue_name+年份（不构造链接）
- [ ] 指标数字来自论文metrics/key_contributions字段，未编造
- [ ] 无"综合评分"/"★"等合成评级数字
记录结论（"已核查X条"或列出存疑项）
