# paper-analysis — 论文深度分析技能

## 使命
从论文数据库中提炼**投资级技术洞察**，而非表面摘要。
目标：让投资经理读完报告后能准确判断——哪个技术路线最有前景、谁在做、有多成熟。

---

## 领域分类策略（重要）

**不使用硬编码的领域 ID 列表。** 先调用 `get_domain_tree()` 获取当前数据库中实际存在的领域分类树和对应 ID，再以此为准进行检索。领域分类由数据平台动态维护，结合人工标注和 AI 补充，比任何静态预设列表都更准确和完整。

获取领域树后，建立 `id → name` 映射，后续检索使用实际 ID，分析时直接读取 `domains[].name` 字段。

> 📖 接口参数详见 `references/api.md`。

---

## 分析工作流策略

### 第一步：领域定位
调用 `get_domain_tree()` 获取领域分类，确认目标方向的 `domain_ids`，建立 ID 到名称的映射。

> 📖 接口参数详见 `references/api.md`。

### 第二步：批量扫描（核心，不要只看少量论文）

**覆盖策略由用户问题决定：**

- **用户关注特定赛道**（"XX技术最新进展"）→ 按 `domain_ids` 分领域深扫，每个方向至少 60 篇，按发布日期降序
- **用户做全面研判** → 全库分页扫描，覆盖近两年，按发布日期排序
- **用户关注特定主题** → 关键词检索，重点读取 `research_problem` 和 `key_contributions` 字段
- **用户描述研究概念** → 向量语义检索，支持中英文混合，能召回与概念相近但措辞不同的论文
- **用户问特定科学家** → 按 `author_name` 检索，无需全库扫描

### 第三步：专题深挖

针对第二步发现的高价值方向，用更精确的关键词或作者名做针对性检索，深度读取 `tech_route`、`key_contributions`、`metrics` 字段。

### 第四步：保存并输出

完成分析后保存为 `paper-analysis` 类别的 artifact，文件名含主题和日期。

## 深度提取策略

### research_problem 字段解读
- `summary`：一句话描述（适合快速分类）
- `detail`：完整问题背景（用 `depth="detail"` 获取）
- 重点关注：提到"商业化"、"工程化"、"规模化"的论文——这些是TRL较高的信号

### 论文字段速查
| 字段 | 说明 | 用途 |
|------|------|------|
| `venue_name` | 发表期刊/会议全称（如 "Physical Review Letters"、"Nature"） | 判断权威性；引用时作为来源 |
| `doi` | DOI 标识符 | 原文链接：`https://doi.org/{doi}` |
| `arxiv_id` | arXiv 预印本 ID | 预印本链接：`https://arxiv.org/abs/{arxiv_id}` |
| `authors` | 作者数组，含 `name` 和 `affiliation` | 机构分析、人才追踪 |
| `domains` | 领域对象数组 `[{id, name, level}]` | 直接读 `d.name`，**无需**再查 `/gold/domains` |
| `research_problem` | 研究问题 `{summary, detail}` | 快速分类和深度分析 |
| `tech_route` | 技术路线 `{summary, detail}` | 路线判断和竞争分析 |
| `key_contributions` | 关键贡献数组 | 突破点和数值指标 |
| `metrics` | 定量指标（qubit_count, fidelity 等） | 硬信号，TRL判断依据 |

### tech_route 字段解读
- 不同技术路线代表不同投资逻辑（原材料/工艺/供应链依赖各异）
- "混合"路线（多技术融合）往往是当前阶段的新兴方向
- 注意技术路线的规模化难点：实验室性能 vs. 商业可重复性

### key_contributions 字段解读
- 有具体数字的贡献是硬信号（定量指标、首次突破某阈值）
- 对比历史最佳（"首次突破..."、"刷新记录..."）= 重要里程碑
- `metrics` 字段：关注性能的关键参数数值（依领域而异）

## 技术成熟度（TRL）判断规则

| TRL | 论文信号 | 投资含义 |
|-----|----------|---------|
| 1-3 | "理论提出"、"首次演示"、单元件实验 | 天使/种子，长周期 |
| 4-5 | "小规模验证"、关键指标首次突破阈值、多单元集成 | Pre-A/A轮 |
| 6-7 | "系统集成"、"规模化验证"、"优势实验" | A/B轮窗口 |
| 8-9 | "商业部署"、"客户验证"、重复性保证 | 成长期 |

> **TRL是工程推断，不是可测量事实。** 报告中所有TRL评估须：①以范围标注（如"TRL 4-5"，不出现精确单值）②附判断依据（引用了哪条具体论文信号）③将实测数据（`metrics` 字段的硬指标，直接引用）与TRL推断明确区分——前者直接引用，后者用"基于[信号X]推断TRL约为N-N"表述。同一论文不同解读可能相差±1级，这是正常的认知不确定性。

## 输出要求

**覆盖维度**（格式根据赛道信息密度自适应——数据充分时用表格，数据不足时如实注明"数据不足，无法评估"，禁止用模糊内容填充格式框架）：

1. **数据覆盖声明**：扫描论文数、覆盖领域、分析深度（detail/summary）

2. **技术路线成熟度对比**：各路线的实测指标 + TRL推断范围 + 推断依据 + 趋势。若赛道技术路线单一，可退化为单段描述；若路线多样，用表格。

3. **近期关键突破**：每条突破包含——成果描述 + 关键数字 + 论文来源（作者、期刊/会议名、年份、DOI或arXiv链接）+ 对TRL的影响。突破条数无下限，有多少写多少，数据不足时如实说明。

4. **主要技术瓶颈**（若数据库中有信号）：瓶颈描述 + 当前实测水平 + 商业化目标 + 预计突破时间。无法从论文中评估时注明"数据不足"。

5. **研究主题热度**：按论文数量排序的研究方向，中国vs国际差距（基于作者机构统计）。

6. **投资关注信号**：最受论文关注的技术拐点、技术路线竞争中的风险点。

7. **引用论文**：报告中提到的所有论文均需列出：
   `[N] 作者等, "论文标题", *期刊/会议名*, 年份. DOI: https://doi.org/{doi}` 或 `arXiv: https://arxiv.org/abs/{arxiv_id}`
   若论文无DOI也无arxiv_id，则列venue_name + 年份。

**数据自查（必填）**
- [ ] 所有TRL均为范围标注（如"TRL 4-5"），无精确单值；每条TRL附推断依据
- [ ] 所有论文来源均有DOI或arXiv链接（最低：venue_name + 年份）
- [ ] 指标数字来自论文metrics/key_contributions字段，未编造
- [ ] 无"综合评分"/"★"等合成评级数字
记录结论（"已核查X条"或列出存疑项）
