| 深度提取策略 | ✅ | 论文字段速查表，abstract 提取要点 |
| TRL评估框架 | ✅ | **亮点**：三维指标（TRL范围+产业化阶段+商业化时间窗口） |
| TRL引用质量要求 | ✅ | 工程化信号判定标准 |
| DOI-标题完整性约束 | ✅ | **亮点**：禁止跨记录拼凑，防止"幽灵引用" |

### 3.4 设计亮点

| 亮点 | 说明 |
|------|------|
| **三维TRL评估** | TRL范围 + 产业化阶段 + 商业化时间窗口，互相印证 |
| **Web交叉验证强制执行** | 论文DB后必须用 search_web 补充工程化/商业化信号 |
| **DOI-标题完整性约束** | 防止跨记录拼凑产生"幽灵引用" |
| **数据库样本局限性声明** | 禁止将内部DB统计推断为全球发表频率 |
| **TRL引用质量门槛** | 只有含工程化信号的论文才能支撑TRL评估 |

### 3.5 关键约束规则

```
TRL 评估规则：
- 必须是范围值（如 TRL 5-7），严禁精确单值
- 商业化时间窗口必须是范围（如 3-5年），禁止精确年份
- 三维指标互相印证，不一致时需重新检查
- 数据不足时明确写"数据不足，无法评估"

引用规则：
- DOI/标题/作者必须来自同一条记录，禁止跨记录拼凑
- 无 DOI 时用 arXiv 链接，两者均空时只列 venue+年份
- 禁止以训练知识编造虚假论文来源

统计规则：
- 内部DB是抽样样本，不等于全球发表分布
- 禁止用DB条数占比推断行业产出占比
```
