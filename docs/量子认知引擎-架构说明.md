# 量子赛道认知引擎 — 架构说明文档

> 更新时间：2026-03-04  
> 版本：v0.5（DeepAgents 0.4.5）

---

## 1. 系统全景

```
用户（投资经理）
       │ POST /deep/stream（SSE 流式）
       ▼
quantum-chat-backend :8001
├── api/deep_research.py          ← 接入层
│       │  注入 skill_files（6个SKILL.md，启动时缓存）
│       ▼
│   Orchestrator Agent            ← 主脑（deepagents.create_deep_agent）
│   技能：/skills/quantum-investment-analysis/
│       │
│       ├── task("paper-researcher", ...)  ┐
│       ├── task("people-intel", ...)      ├── 并行发出（同一 LLM 响应中）
│       └── task("news-market", ...)       ┘
│
├── SubAgent: paper-researcher    技能：/skills/paper-research/
│   数据源：量子API论文库 (460篇顶刊顶会)
│
├── SubAgent: people-intel        技能：/skills/people-intel/
│   数据源：量子研究员档案 (800+人)
│
└── SubAgent: news-market         技能：/skills/news-market/
    数据源：Tavily 实时互联网搜索

产物归档：research-artifacts/YYYY-MM-DD/*.md
```

## 2. 各 Agent 角色与能力

### 2.1 Orchestrator（量子赛道认知引擎）

**角色**：主脑规划者，不直接查数据，只协调子 Agent 并综合输出。

**核心行为**：
1. 收到用户请求 → `write_todos` 拆解任务
2. 若为全面研判 → **同时**发出三个 `task()` 调用（并行，不串行）
3. 收到三份子 Agent 报告 → 综合提炼，生成最终投资研判报告

**输出规范（全面研判）**：
- 最低 2500 字
- 必须包含：执行摘要 / 技术TRL表格 / 人才h-index表 / 融资全表 / 投资评分矩阵
- 所有数据保留来源和时效，不压缩

---

### 2.2 paper-researcher（量子论文分析师）

**角色**：从 460 篇近一年顶刊顶会论文中提炼投资级技术洞察。

| 属性 | 内容 |
|------|------|
| 工具 | `get_domain_tree` `batch_scan_papers` `search_papers` `analyze_quantum_theme` `save_research_artifact` |
| 数据源 | 量子引擎 API (http://47.110.226.140:8080)，460篇论文 |
| 技能文件 | `/skills/paper-research/SKILL.md` |
| 产物类别 | `paper-analysis` |

**工作流**：
1. `get_domain_tree()` → 获取领域ID体系
2. `batch_scan_papers(pages=2)` → 批量覆盖120篇，获取 detail 字段
3. `search_papers(domain_ids=..., depth="detail")` → 专题深挖
4. 分析TRL、保真度演进、中国vs国际差距
5. `save_research_artifact(category="paper-analysis")` → 保存 .md 产物

**输出要点**：
- TRL对比表（超导/离子阱/光量子/中性原子/量子纠错）
- 关键突破列表（含具体数字 + 论文引用 ≥5篇）
- 投资信号：技术路线热度、拐点判断

---

### 2.3 people-intel（量子人才情报分析师）

**角色**：系统绘制中国量子赛道人才图谱，识别产业化关键人物。

| 属性 | 内容 |
|------|------|
| 工具 | `get_institution_researchers` `search_researchers` `get_researcher_detail` `save_research_artifact` |
| 数据源 | 量子引擎 API，800+ 研究员档案 |
| 技能文件 | `/skills/people-intel/SKILL.md` |
| 产物类别 | `people-intel` |

**工作流**：
1. 按五大谱系机构逐一扫描：中科大 → 浙大 → 清华 → 北京量子院 → 南科大
2. 按研究方向补充（量子纠错/离子阱/中性原子）
3. TOP PI 深度画像：机构 + h-index + 近3年论文 + 产业化关联
4. `save_research_artifact(category="people-intel")` → 保存产物

**五大学术谱系**（背景知识）：

| 谱系 | 代表人物 | 机构 | 主攻方向 | 产业化 |
|------|----------|------|----------|--------|
| 潘建伟系 | 潘建伟、陆朝阳、朱晓波 | 中科大 | 量子通信、光量子、超导 | 国盾量子、本源量子 |
| 王浩华系 | 王浩华 | 浙大 | 超导量子 | 阿里合作 |
| 段路明系 | 段路明 | 清华/MIT | 离子阱、量子纠错 | — |
| 俞大鹏系 | 俞大鹏 | 南科大/北京量子院 | 超导量子 | 国家队平台 |
| 薛其坤系 | 薛其坤 | 清华/北京量子院 | 拓扑量子 | — |

**输出要点**：
- 核心科学家表（含 h-index 列，量化指标）
- 机构实力对比表
- 五大谱系分别概述，缺失标注 `[数据缺失]`
- 产业化人物标注具体公司名

### 2.4 news-market（量子市场情报分析师）

**角色**：从实时互联网中捕捉投资信号，精确到融资金额和政策预算。

| 属性 | 内容 |
|------|------|
| 工具 | `search_quantum_funding` `search_quantum_news` `search_quantum_policy` `search_quantum_companies` `save_research_artifact` |
| 数据源 | Tavily 实时互联网 |
| 技能文件 | `/skills/news-market/SKILL.md` |
| 产物类别 | `market-intel` |

**工作流**：
1. 融资情报：中/英文双向搜索近期融资事件
2. 政策情报：十五五专项、各地量子产业政策预算数字
3. 市场规模：最新市场预测数字
4. 竞争动态：Google/IBM/本源/国盾等关键公司最新动向
5. `save_research_artifact(category="market-intel")` → 保存产物

**输出要点**：
- 融资事件完整 Markdown 表格（公司/金额/轮次/投资方/时间/来源）
- 政策预算数字（附来源文件）
- 来源不明标注 `[来源未确认]`，查不到标注 `[未查到]`

## 3. Skills 系统（渐进式加载）

本系统 Skills 遵循 [Agent Skills Specification (agentskills.io)](https://agentskills.io/specification)，实现与 Claude Code 相同的**三级渐进式加载**机制。

### 3.1 三级加载机制（Progressive Disclosure）

```
第1级（启动时，~100 tokens/skill）
  ↓ deepagents 读取每个 SKILL.md 的 YAML frontmatter
  ↓ 只提取 name + description 构成 Skills System 注入 system prompt

第2级（用户提问时，按需激活）
  ↓ Agent 检查 description 是否与当前任务匹配
  ↓ 若匹配，读取完整 SKILL.md 正文（工作流 + 模板 + 规则）

第3级（任务执行中，按需加载）
  ↓ 若 SKILL.md 正文中引用了 references/ scripts/ assets/ 等子文件
  ↓ Agent 按需读取这些文件（当前版本仅有单文件，待扩展）
```

**核心优势**：相比将所有工作流都写死在 system_prompt（每次消耗数千 tokens），Skills 系统在不需要时几乎零成本，仅在相关任务激活时才占用 context。

### 3.2 Skills 文件结构

```
quantum-chat-backend/skills/
├── quantum-investment-analysis/  ← Orchestrator 专属
│   └── SKILL.md                  编排策略、评分矩阵、调度规则
├── paper-research/               ← paper-researcher 专属
│   ├── SKILL.md                  领域ID索引、TRL判断规则、Batch扫描策略
│   └── references/
│       └── api.md                论文库 API 详细参数说明
├── people-intel/                 ← people-intel 专属
│   ├── SKILL.md                  五大谱系背景、机构表、查询工作流
│   └── references/
│       └── api.md
├── news-market/                  ← news-market 专属
│   ├── SKILL.md                  Tavily查询组合、公司追踪清单
│   └── references/
│       └── api.md
├── quantum-research-workflow/    ← 研究工作流编排功能
│   └── SKILL.md
└── github-trends-ossinsight-api/ ← GitHub 趋势数据分析
    └── SKILL.md
```

### 3.3 Skill 隔离规则

- Orchestrator 只看到 `/skills/quantum-investment-analysis/`
- 各子 Agent 只看到自己的 skill 路径
- **主 Agent 的 skills ≠ 子 Agent 的 skills**（完全隔离）

### 3.4 SKILL.md 格式规范

```markdown
---
name: skill-name              # 小写字母+连字符，必须与目录名一致
description: 一句话描述...    # 1-1024字符，精准描述触发条件
license: MIT
---

# skill-name — 技能标题

## 使命
...(正文内容)
```

**关键**：`description` 字段是 Agent 决定是否激活 skill 的唯一依据，务必清晰具体。YAML 仅保留 `name`、`description`、`license`，无需其他元数据字段。

## 4. 请求处理流程

### 4.1 全面投资研判（全量模式）

**示例问题**：`"请对量子计算赛道做全面投资研判，包括技术、人才和市场三个维度"`

```
① POST /deep/stream → api/deep_research.py
   ↓ 注入 skill_files（缓存，不重复 IO）

② Orchestrator 激活 quantum-investment-analysis skill
   ↓ write_todos 拆解：技术/人才/市场三条线
   ↓ 一次 LLM 响应发出三个并行 task()

③ 三条流水线同时运行（预期约 3-5 分钟）：
   paper-researcher (约5min)        people-intel (约2min)     news-market (约2min)
   get_domain_tree()                 get_institution_          search_quantum_funding
   batch_scan_papers(pages=2)        researchers × 5           search_quantum_policy
   search_papers(detail)             search_researchers        search_quantum_companies
   analyze_quantum_theme             get_researcher_detail     search_quantum_news
   save_artifact("paper-analysis")   save_artifact             save_artifact

④ Orchestrator 收到三份报告
   ↓ 综合提炼 → 生成 ≥2500字 最终报告
   结构：执行摘要 / 技术研判 / 人才洞察 / 市场信号 / 投资建议

⑤ 每个 token 实时 SSE 推流到前端
   {type:"subagent_token", agent:"paper-researcher", content:"..."}
   {type:"token", agent:"quantum-orchestrator", content:"..."}
   {type:"done", thread_id:"xxx"}

⑥ 产物写入 research-artifacts/2026-03-04/
   paper-analysis-*.md / people-intel-*.md / market-intel-*.md
```

### 4.2 按需单维度查询（轻量模式）

Orchestrator 会根据问题性质**只调用相关子 Agent**，而非默认调用三个：

| 问题类型 | 调用 Agent |
|----------|-----------|
| "超导量子最新进展是什么？" | paper-researcher only |
| "本源量子最近有什么融资动态？" | news-market only |
| "潘建伟团队有哪些核心人物？" | people-intel only |
| "本源量子怎么样？" | people-intel + news-market（公司尽调模式）|
| "给我做全面量子赛道投资研判" | 三个 Agent 并行 |
