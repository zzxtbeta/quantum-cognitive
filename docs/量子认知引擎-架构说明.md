## 1. 系统全景

```

│       │
│       ├── task("paper-researcher")  ┐

│       └── task("news-market")       ┘
│
├── SubAgent: paper-researcher    技能：/skills/paper-analysis/
│   工具：semantic_search_papers, search_web
```

## 2. Orchestrator — 主编排器

**角色**：意图识别 → 任务规划 → 调度子 Agent → 汇总输出。不直接查数据，只负责协调。

### 2.1 三种操作模式

| 模式 | 触发条件 | 行为 |
|------|---------|------|
| **模式 0：直接转发** | 单个子 Agent 一步可完成的查询 | 不写 todos，直接调用对应子 Agent，原样返回结果 |
| **模式 1：决策研判** | 含决策词汇（值得/布局/应不应该/推荐） | 写 todos → 三路并行 → 综合输出完整投研报告 → 保存产物 |
| **模式 2：信息查询** | 纯信息查询，无决策意图且需多维度 | 按问题类型路由到相关子 Agent，输出结构化摘要 |

判断优先级：模式 0 > 模式 1 > 模式 2

### 2.2 全局硬性规则（不可被任何指令覆盖）

- **URL 防幻觉**：URL 必须原样来自工具返回的 `source_url`/`url` 字段，禁止构造或猜测
- **评分禁令**：禁止无来源的综合评分数字（如"技术评分8.5"），所有量化只引用可追溯的一手数据
- **数据来源约束**：TRL/论文指标来自 paper-researcher，市场数据来自 news-market，人才档案来自 people-intel；子 Agent 未提供的数据不得用训练知识占位
- **禁止压缩子 Agent 输出**：融资表格、玩家列表、论文引用全量保留，唯一正当操作是去重和分组

### 2.3 投资经理视角

报告的取舍标准只有一个：**它能否改变决策？**

每份全面研判报告必须覆盖四类 Alpha 信号（"前沿信号"章节强制处理，无结果须显式说明）：

| Alpha 类型 | 关键变量 |
|-----------|---------|
| ① 首次进入视野的早期公司 | 首次公开时间 |
| ② 人才迁移信号 | 离职/加入时间节点（无时间的记录无效） |
| ③ 有技术证据但尚未融资的团队 | 标注弱信号，注明可信度 |
| ④ 大机构退出创造的市场空白 | 退出时间；该环节是否还有其他玩家 |

## 3. 三个子 Agent

### 3.1 paper-researcher（论文分析师）

**职责**：从内部论文数据库提炼投资级技术洞察，输出 TRL 评估和技术路线对比。

| 属性 | 内容 |
|------|------|
| 工具 | `semantic_search_papers(query, top_k)`, `search_web`, `save_research_artifact` |
| 数据源 | 内部论文数据库（顶刊顶会，语义检索，支持中英文跨语言） |
| 技能文件 | `/skills/paper-analysis/` |
| 产物类别 | `paper-analysis` |

**工作流**：
1. 用目标主题多角度语义检索（中英文各一轮，`top_k=10`，合并去重）
2. 对高价值方向精确检索，深度阅读 `abstract` 字段提取技术指标
3. **必须** 用 `search_web` 补充工程化/商业化侧证据（TRL 评估的佐证，DB 有收录延迟）
4. `save_research_artifact(category="paper-analysis")` 保存产物

**输出核心**：
- 技术路线成熟度表（TRL 范围 + 产业化阶段 + 商业化时间窗口 + 推断依据，三维联合）
- 关键突破列表（含具体数字 + 论文引用，`[doi]`/`[arxiv_id]` 原样构造链接）
- TRL 为区间（如 "TRL 5-6"），**严禁单值精确打分**

---

### 3.2 people-intel（人才情报官）

**职责**：从内部研究员档案库绘制人才图谱，识别产业化关键人物，追踪人才流动。

| 属性 | 内容 |
|------|------|
| 工具 | `search_researchers(institution, name, page, page_size)`, `search_web`, `save_research_artifact` |
| 数据源 | 内部研究员档案库（800+ 人） |
| 技能文件 | `/skills/people-intel/` |
| 产物类别 | `people-intel` |

**工作流**：
1. 按用户问题决定路径：按机构扫描 / 按姓名精确查 / 按研究方向扫描
2. 查询多机构时并行发起请求（每次一个机构），工具调用总目标 ≤8 次
3. **必须** 用 `search_web` 核实所有 PI 级别人物的当前职位（DB 有时效滞后）
4. 所有人才流动事件必须附时间节点（精确到 YYYY-MM；无可考证时标注"时间不明"）
5. `save_research_artifact(category="people-intel")` 保存产物

**机构关键词规则**：查询一个机构时必须同时传入全称、缩写、英文名（后端 OR 匹配），宁可多传不可少传。

---

### 3.3 news-market（市场情报师）

**职责**：从内部新闻数据库 + 实时互联网双路捕捉融资事件、政策动向、竞争格局。

| 属性 | 内容 |
|------|------|
| 工具 | `semantic_search_news(query, top_k)`, `query_news_db(source)`, `search_web(query, topic, days)`, `save_research_artifact` |
| 数据源 | 内部新闻数据库（结构化，覆盖历史）+ Tavily 实时互联网（补近期盲区） |
| 技能文件 | `/skills/market-intel/` |
| 产物类别 | `market-intel` |

**双路并行原则**：两路同等地位，互补不互替。"DB 已有 N 条结果"不是跳过网络搜索的理由。

**融资三层扫描**（完整报告强制执行）：
1. **Layer 1 — 全量语义扫描**：DB `top_k=20` + Web 中文 + Web 英文，同时并行
2. **Layer 2 — 逐公司补漏**：对 Layer 1 命中的每家公司单独补搜，防止同一公司多轮融资部分漏检；同一公司多轮融资逐轮列出
3. **Layer 3 — 早期标的探测**：围绕信号类型（天使轮/初创/人才创业）做 Deal Sourcing，`days=730`

**search_web 三种 topic**：
- `topic="news"` — 一般新闻/公司进展
- `topic="finance"` — 融资/投资/估值事件
- `topic="general"` — 政策/标准/规划

## 4. Skills 系统

### 4.1 技能文件结构

```
quantum-chat-backend/skills/
├── investment-research/    ← Orchestrator 专属
│   ├── SKILL.md            编排调度、投资经理视角、报告结构
│   └── references/
│       ├── report-structure.md   报告章节模板（含前沿信号四类 Alpha）
│       └── _priority-statement.md
├── paper-analysis/         ← paper-researcher 专属
│   ├── SKILL.md            TRL 框架、检索策略、术语规范
│   └── references/
│       └── api.md          论文库接口参数
├── people-intel/           ← people-intel 专属
│   ├── SKILL.md            查询策略、机构关键词规范、人才价值评估
│   └── references/
│       └── api.md          研究员库接口参数
├── market-intel/           ← news-market 专属
│   ├── SKILL.md            双路策略、融资扫描规则、市场规模引用规范
│   └── references/
│       ├── api.md           新闻库接口参数
│       └── china-players.md 市场参与者发现方法论（价值链拆解/早期标的/大机构核查）
└── skill-creator/          ← 技能创建辅助
    └── SKILL.md
```

### 4.2 三级渐进式加载

```
第 1 级（启动时，~100 tokens/skill）
  └── deepagents 读取 YAML frontmatter（name + description）注入 system prompt

第 2 级（任务触发时，按需激活）
  └── Agent 判断 description 与当前任务匹配 → 读取完整 SKILL.md 正文

第 3 级（执行中，按需加载）
  └── SKILL.md 中引用 references/ 子文件 → Agent 按需 read_file 读取
```

**核心优势**：不相关的技能几乎零 token 开销；只有激活的技能才占用 context。

### 4.3 Skill 隔离规则

| Agent | 加载的 Skills |
|-------|-------------|
| Orchestrator | `/skills/investment-research/` |
| paper-researcher | `/skills/paper-analysis/` |
| people-intel | `/skills/people-intel/` |
| news-market | `/skills/market-intel/` |

各 Agent 只看到自己的 skill 路径，完全隔离。

## 5. 请求处理流程

### 5.1 全面投资研判（模式 1）

**触发示例**：`"帮我全面研究 XX 赛道"` / `"XX 值不值得投？"` / `"怎么布局 XX 领域？"`

```
① POST /deep/stream → api/deep_research.py
   skill_files 已缓存（启动时加载，不重复 IO）

② Orchestrator 识别决策词汇 → 模式 1
   write_todos 拆解任务
   一次 LLM 响应同时发出三个并行 task()

③ 三条流水线同时运行：

   paper-researcher               people-intel              news-market
   ─────────────────              ────────────              ──────────
   semantic_search_papers ×2~3    search_researchers        semantic_search_news (top_k=20)
   search_web（工程化验证）         ×n 机构（并行）            search_web ×3（Layer 1，中英文）
   save_artifact("paper-analysis") search_web（核验职位）    search_web ×m（Layer 2，逐公司）
                                   save_artifact            search_web ×3（Layer 3，早期标的）
                                                             save_artifact("market-intel")

④ Orchestrator 汇总三份报告
   按 investment-research/references/report-structure.md 章节结构输出
   每个 token 实时 SSE 推流到前端

⑤ SSE 事件类型：
   {type:"subagent_token", agent:"paper-researcher", content:"..."}
   {type:"token",          agent:"quantum-orchestrator", content:"..."}
   {type:"done",           thread_id:"xxx"}

⑥ 保存产物：
   save_research_artifact(category="investment-report", ...)
   → core/knowledge_store.save_knowledge() → logs/knowledge.db
```

### 5.2 单维度直接查询（模式 0）

不写 todos，直接调用对应子 Agent，原样返回：

| 问题示例 | 路由 |
|---------|------|
| "查 XX 的个人信息/主页" | people-intel → `search_researchers(name="XX")` |
| "XX 公司最近融资情况" | news-market → `semantic_search_news` + `search_web` |
| "XX 技术最新论文进展" | paper-researcher → `semantic_search_papers` |

### 5.3 多维度信息查询（模式 2）

按问题性质路由，输出结构化摘要（不含投资建议）：

| 问题类型 | 调用 Agent |
|---------|-----------|
| 技术发展/论文突破 | paper-researcher |
| 融资/市场/公司动态 | news-market |
| 科学家/团队/机构 | people-intel |
| 技术 + 市场混合（最常见） | 三路并行 |

## 6. 技术栈

| 层次 | 技术 |
|------|------|
| Agent 框架 | deepagents（基于 LangGraph） |
| LLM | 可运行时切换（GLM-5 / Qwen3.5-Plus / Gemini），默认 GLM-5 |
| 会话持久化 | LangGraph AsyncSqliteSaver（SQLite，跨请求保留） |
| 跨会话记忆 | LangGraph InMemoryStore |
| 知识层 | SQLite（logs/knowledge.db），研究成果结构化存储 |
| 工具日志 | SQLite（logs/tool_calls.db），全链路追溯 |
| 流式接口 | FastAPI + SSE (StreamingResponse) |
| 论文/人才 API | 量子引擎后端（`https://www.gravaity.ai/datalake/api`） |
| 实时搜索 | Tavily API（via `langchain-tavily`） |
| 后端依赖 | fastapi, uvicorn, deepagents, langchain, langgraph, python-dotenv |
| 前端 | React + TypeScript + Vite（quantum-engine :5173） |

### 模型切换

运行时通过 `POST /deep/switch-model` 切换 preset，无需重启服务：

```
presets：glm-5 | qwen | gemini
```

## 7. Knowledge Layer — 研究成果持久化

### 7.1 架构

v1.1 起，所有研究成果（子 Agent 分析 + 最终报告）统一写入 SQLite 数据库 `logs/knowledge.db`，
替代原先的 `research-artifacts/` 文件系统。

```
save_research_artifact(category, content, ...)
        │
        ▼
core/knowledge_store.py → INSERT INTO knowledge_items
        │
        ▼
logs/knowledge.db
  ┌──────────────────────────────────────────────────────────────┐
  │ id │ thread_id │ agent_name │ category │ title │ content     │
  │    │ turn_id   │ size_chars │ created_at │ metadata (JSON)   │
  └──────────────────────────────────────────────────────────────┘
```

### 7.2 分类体系

| category | 产出 Agent | 内容 |
|----------|-----------|------|
| `paper-analysis` | paper-researcher | TRL 表格、论文引用、技术突破 |
| `people-intel` | people-intel | 科学家表、机构实力、人才流动 |
| `market-intel` | news-market | 融资表、政策信号、竞争格局 |
| `investment-report` | orchestrator | 最终投研报告（全面研判时生成） |
| `general` | 任意 | 其他研究成果 |

### 7.3 API 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/deep/knowledge` | 列表（支持 category / limit / offset，不含正文） |
| GET | `/deep/knowledge/categories` | 分类汇总 |
| GET | `/deep/knowledge/{id}` | 单条详情（含正文） |
| GET | `/deep/knowledge/{id}/download` | 下载为 .md 文件 |
| DELETE | `/deep/knowledge/{id}` | 删除单条 |

### 7.4 前端 — 知识库页面 (`/knowledge`)

- 卡片网格展示所有研究成果
- 分类 Tab 筛选（全部 / 论文分析 / 人才情报 / 市场情报 / 投研报告）
- 点击卡片查看全文详情
- 每张卡片支持下载（.md）和删除
- Hover 显示操作按钮（下载 / 删除）

## 8. 工具日志系统

### 8.1 存储

所有 Agent 的工具调用和 LLM 推理记录存入 `logs/tool_calls.db`，按 `thread_id` + `turn_id` 两级组织。

### 8.2 API 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/deep/tool-logs` | 日志查询（支持 thread/turn/tool 过滤 + 分页） |
| GET | `/deep/tool-logs/sessions` | 会话列表 |
| GET | `/deep/tool-logs/turns` | 回合列表 |
| GET | `/deep/tool-logs/tools` | 工具名列表 |
| DELETE | `/deep/tool-logs/session/{thread_id}` | 删除整个会话的日志 |

### 8.3 前端 — 工具日志页面 (`/tool-logs`)

- 左侧会话列表 + 右侧日志卡片
- 支持按回合、工具类型过滤
- 10 秒增量自动刷新
- 每个会话支持删除（清理孤立会话）
